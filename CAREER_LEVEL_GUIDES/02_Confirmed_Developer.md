# Manager's Guide: Confirmed Developer

A comprehensive guide for managing Confirmed Developers and helping them grow toward Senior level.

---

## ðŸŽ¯ Overview: The Confirmed Developer

**What defines this level:**
Confirmed Developers are **independent, consistent contributors** who deliver quality work with minimal guidance. They own features end-to-end and are expanding their technical and collaborative scope.

**Key characteristics:**
- Self-sufficient in familiar domains
- Consistently delivers quality work
- Takes ownership of features and bugs
- Proactively communicates
- Beginning to mentor others
- Growing technical depth and breadth

**Typical tenure:** 2-4 years of professional experience

## ðŸŽ“ Development Priorities

### 1. **Technical Ownership** (Engineering Craft)

**What it looks like:**
- Owns features from design through production
- Handles complexity with minimal guidance
- Makes good technical trade-offs
- Writes maintainable, well-tested code
- Debugs issues independently

**How to develop this:**
- Assign complete features, not just tasks
- Let them make technical decisions (with oversight)
- Give ownership of production components
- Encourage them to write technical specs
- Have them lead technical discussions

**Warning signs they're not there yet:**
- Still needs frequent check-ins on approach
- Requires significant rework in code review
- Struggles with unfamiliar technical areas
- Doesn't anticipate edge cases

---

### 2. **Consistent Quality** (Engineering Craft)

**What it looks like:**
- Code is clean, readable, maintainable
- Comprehensive test coverage
- Few bugs in production
- Good code review feedback incorporation
- Thoughtful API design

**How to develop this:**
- Set quality standards explicitly
- Deep code review discussions
- Encourage refactoring
- Teach design patterns
- Review their code reviews

**Warning signs they're not there yet:**
- Quality varies significantly project to project
- Skips testing or writes superficial tests
- Code reviews result in many changes
- Production bugs are frequent

---

### 3. **Cross-Team Collaboration** (Soft Skills)

**What it looks like:**
- Works effectively with Product, Design, other eng teams
- Communicates proactively
- Asks clarifying questions
- Provides status updates without prompting
- Handles feedback constructively

**How to develop this:**
- Assign projects requiring cross-team coordination
- Have them present in team meetings
- Encourage written documentation
- Model good collaboration behaviors
- Debrief after cross-team interactions

**Warning signs they're not there yet:**
- Stays siloed in their own work
- Needs prompting to communicate
- Surprises stakeholders with delays
- Struggles to explain technical concepts

---

### 4. **Proactive Problem Solving** (Soft Skills)

**What it looks like:**
- Identifies issues before they escalate
- Proposes solutions, not just problems
- Takes initiative on technical debt
- Asks for help appropriately
- Unblocks themselves

**How to develop this:**
- Encourage "bring solutions, not just problems"
- Recognize initiative publicly
- Give space to try their ideas
- Support their improvement proposals
- Create psychological safety to take risks

**Warning signs they're not there yet:**
- Waits to be told what to do
- Escalates issues immediately without attempting solutions
- Doesn't surface problems until late
- Needs significant hand-holding

---

### 5. **Mentoring Basics** (Leadership)

**What it looks like:**
- Helps Junior developers through pair programming
- Provides thoughtful code review feedback
- Shares knowledge in team channels
- Answers questions patiently
- Documents learnings for others

**How to develop this:**
- Pair them with Junior developers
- Have them review others' code
- Encourage them to give tech talks
- Ask them to document processes
- Model effective mentoring

**Warning signs they're not there yet:**
- Avoids helping others
- Code review feedback is terse or unhelpful
- Doesn't share knowledge
- Impatient with questions

## ðŸ’¬ Career Conversations

### First Conversation: Understanding Their Path

**Within first month of managing them:**

**Your opening:**
"I'd love to understand where you are in your career journey and what you're working toward. Let's start with: What are you most excited about learning right now?"

**Key questions to ask:**

**About their current work:**
- What parts of your work energize you most?
- What do you find challenging right now?
- Where do you feel you're growing?
- What would you like to be doing more of?

**About their aspirations:**
- Where do you see yourself in 1-2 years?
- What does Senior Developer mean to you?
- What skills do you want to develop?
- Are there any areas you'd like to explore?

**About support:**
- What kind of support helps you learn best?
- How much autonomy do you prefer?
- What feedback style works for you?
- What would make you more effective?

**What you're assessing:**
- Do they want to grow toward Senior level?
- Are they interested in technical depth or leadership?
- Do they understand what growth looks like?
- What motivates them?

## ðŸŽ¯ Example Development Goals

### Goal Set for Early Confirmed Developer (Recently promoted from Junior)

**Focus: Building consistency and confidence**

**Goal 1: Technical Ownership** (Engineering Craft)
"Own the delivery of 3 medium-complexity features end-to-end this quarter (from technical design through production) with minimal guidance, demonstrating ability to handle 2x the complexity of previous work"

**Success criteria:**
- Write technical specs for each feature
- Complete with <5 blocking code review comments per feature
- Zero critical production bugs within 30 days of launch
- Deliver within estimated timeline

---

**Goal 2: Cross-Team Collaboration** (Soft Skills)
"Successfully collaborate with Product and Design teams on upcoming dashboard redesign, demonstrating proactive communication and partnership"

**Success criteria:**
- Attend kickoff and weekly syncs
- Proactively surface technical constraints early
- Provide weekly status updates without prompting
- Positive feedback from PM and Designer

---

**Goal 3: Knowledge Sharing** (Leadership)
"Build mentoring skills by helping Junior developers through pair programming and code review feedback"

**Success criteria:**
- Pair program with Junior developer 2x per month
- Provide thorough code review feedback on 10+ PRs
- Receive positive feedback from Junior developers

---

### Goal Set for Senior-Track Confirmed Developer (2+ years at level)

**Focus: Demonstrating Senior-level capabilities**

**Goal 1: Technical Leadership** (Engineering Craft)
"Lead the architecture and implementation of the notification system refactor (cross-team scope), demonstrating Senior-level technical ownership"

**Success criteria:**
- Design system architecture with Staff engineer guidance
- Present design to architecture review and incorporate feedback
- Successfully migrate 3 services with zero downtime
- Create reusable patterns adopted by other teams

---

**Goal 2: Cross-Org Influence** (Soft Skills)
"Build relationships and collaborate effectively with Platform, Mobile, and Backend teams on notification contracts"

**Success criteria:**
- Lead technical sync meetings across teams
- Navigate conflicting requirements and reach consensus
- Maintain clear communication with all stakeholders
- Deliver on commitments to all teams

---

**Goal 3: Mentoring Impact** (Leadership)
"Mentor 1 Confirmed developer to accelerate their growth toward Senior level"

**Success criteria:**
- Have monthly career development conversations
- Provide regular code review feedback
- Help them successfully deliver 2 complex projects
- Developer reports growth in [specific skills]

## ðŸ—ï¸ Stretch Projects for Confirmed Developers

### For Building Technical Depth

**Project type: Complex feature with unfamiliar technology**

**Why it's a stretch:**
- Forces learning new technical domain
- Requires independent research
- Tests problem-solving without a playbook

**Example:**
"Implement real-time notifications using WebSockets (technology they haven't used)"

**Your support:**
- Point to resources and examples
- Check in on approach early
- Available for questions, but don't solve it for them
- Debrief learnings after

---

### For Building Technical Breadth

**Project type: Cross-stack or cross-service work**

**Why it's a stretch:**
- Expands understanding beyond their usual domain
- Requires understanding system interactions
- Builds empathy for other teams' constraints

**Example:**
"Add a new API endpoint and connect it through the full stack: database â†’ backend â†’ API â†’ frontend"

**Your support:**
- Connect them with experts in each area
- Help them understand each layer
- Encourage them to ask questions across teams

---

### For Building Collaboration Skills

**Project type: Feature requiring heavy cross-team coordination**

**Why it's a stretch:**
- Must work with Product, Design, and other engineering teams
- Requires clear communication and status updates
- Tests stakeholder management

**Example:**
"Partner with Design team to implement new user onboarding flow"

**Your support:**
- Make introductions to stakeholders
- Coach on communication strategies
- Debrief after key meetings
- Provide feedback on collaboration style

---

### For Building Leadership

**Project type: Leading a small initiative or process improvement**

**Why it's a stretch:**
- First taste of technical leadership
- Requires influencing without authority
- Tests facilitation skills

**Example:**
"Lead the team's code quality improvement initiative over the next quarter"

**Your support:**
- Help them scope the initiative
- Give them autonomy in execution
- Provide coaching on influencing others
- Recognize their leadership publicly

## âš ï¸ Common Challenges & Solutions

### Challenge 1: "Stuck" at Confirmed Level

**Symptoms:**
- Been at level 2-3+ years
- Doing good work but not expanding scope
- Seems comfortable, not pushing boundaries

**Possible causes:**
- Lack of clear path or goals
- No Senior-level opportunities available
- Prefer current scope (might not want promotion)
- Fear of failure on harder work

**Solutions:**
- Have honest conversation about career goals
- Create explicit development plan
- Assign stretch projects intentionally
- Provide more feedback and coaching
- Consider if they're happy here (that's okay!)

---

### Challenge 2: Inconsistent Performance

**Symptoms:**
- Great on some projects, struggles on others
- Quality varies significantly
- Unpredictable

**Possible causes:**
- Certain types of work are outside comfort zone
- Morale or engagement issues
- External factors (life circumstances)
- Unclear expectations

**Solutions:**
- Identify patterns: what's different between strong and weak work?
- Address skill gaps with focused development
- Check in on morale and engagement
- Clarify expectations explicitly
- Adjust project assignments to build confidence

---

### Challenge 3: Technical Focus, Weak Collaboration

**Symptoms:**
- Strong technically but poor communication
- Works in isolation
- Stakeholders complain they're "hard to work with"
- Surprises people with delays or changes

**Possible causes:**
- Values technical work over "soft" skills
- Introverted personality
- Doesn't see collaboration as part of the job
- Lacks skills/models for good collaboration

**Solutions:**
- Explicit feedback on collaboration gaps
- Set communication expectations clearly
- Assign projects requiring collaboration
- Model and coach effective communication
- Celebrate collaboration successes
- Make it clear: collaboration is part of performance, not optional

---

### Challenge 4: Wants Senior, But Not Ready

**Symptoms:**
- Asking about promotion frequently
- Sees self as Senior-level already
- Frustrated about timeline

**Possible causes:**
- Overestimating current performance
- Doesn't understand Senior level expectations
- Comparing to others unfairly
- Time-based expectation ("I've been here X years")

**Solutions:**
- Show them the framework explicitly
- Give concrete examples of gaps
- Create measurable development plan
- Set realistic timeline (12-18+ months)
- Provide stretch opportunities to demonstrate readiness
- Be direct and honest

---

### Challenge 5: High Performer, Retention Risk

**Symptoms:**
- Clearly ready for Senior level
- Company has budget/promotion constraints
- Getting external interest
- Showing signs of disengagement

**Possible causes:**
- Legitimate career frustration
- Feeling undervalued
- Lack of challenge
- Better opportunities elsewhere

**Solutions:**
- Be transparent about promotion timeline
- Advocate fiercely to leadership
- Provide other recognition (comp, title variations if possible)
- Give them highest-impact projects
- Discuss their career goals openly
- Be honest about what you can and can't control

### Meets Expectations (Solid Confirmed Developer)

**Engineering Craft:**
- Consistently delivers quality features independently
- Code reviews require minimal changes
- Tests are comprehensive
- Production bugs are rare
- Handles complexity appropriate to level

**Soft Skills:**
- Communicates proactively with team and stakeholders
- Collaborates effectively across functions
- Takes initiative on improvements
- Receptive to feedback

**Leadership:**
- Helps Junior developers regularly
- Provides useful code review feedback
- Contributes to team processes
- Shares knowledge

**Overall:** Reliable, consistent contributor. Team would miss them if they left.

---

### Exceeds Expectations (Senior-Track)

**Engineering Craft:**
- Handles Senior-level complexity
- Makes architecture decisions that are consistently sound
- Code is exemplary, cited as reference for others
- Technical depth in their domain
- Solves problems creatively

**Soft Skills:**
- Proactively manages stakeholders
- Explains technical concepts clearly to non-technical partners
- Identifies and solves problems before escalation
- Influences technical direction

**Leadership:**
- Actively mentors others with measurable impact
- Improves team processes
- Takes on leadership responsibilities
- Viewed as go-to person by team

**Overall:** Operating at next level consistently. Promotion conversation is appropriate.

- Frequent production bugs
- Code requires significant rework
- Misses deadlines without communication
- Quality is inconsistent
- Struggles with complexity

**Soft Skills:**
- Needs prompting to communicate
- Doesn't ask clarifying questions
- Works in isolation
- Defensive about feedback

**Leadership:**
- Doesn't help others
- Negative influence on team
- Resists process improvements

**Overall:** Not meeting Confirmed level expectations. Needs performance improvement plan.

## ðŸ“š Resources & Examples

### Sample Technical Specs They Might Write

**For a Confirmed Developer, good technical specs include:**
- Clear problem statement
- Proposed solution with alternatives considered
- API contracts or data models
- Edge cases and error handling
- Testing strategy
- Rollout plan

**What to coach on:**
- Deeper analysis of trade-offs
- Consider more edge cases
- Think about monitoring and observability
- Plan for failure scenarios

---

### Sample Projects at Confirmed Level

**Well-scoped projects:**
- Implement user profile editing feature (medium complexity)
- Refactor legacy authentication module with tests
- Build internal admin dashboard
- Optimize database queries for performance
- Integrate third-party API (e.g., payment processor)

**Too small (Junior-level):**
- Fix individual bugs
- Update UI text
- Add a simple form field

**Too large (Senior-level):**
- Design entire microservice architecture
- Lead multi-quarter cross-team initiative
- Migrate entire system to new technology

---

### Effective Code Review Feedback from Confirmed Developer

**Good example:**
```
Looking at the error handling here, I'm concerned about the case where 
the API returns a 500. Right now we'd show a generic error, but the user 
won't know if they should retry or contact support. Could we differentiate 
between server errors and validation errors?

Also, we might want to add logging for this failure case to help debugging. 
What do you think?
```

**Why it's good:**
- Specific concern identified
- Explains the impact
- Suggests a solution
- Asks question (collaborative tone)
- Considers user experience and debugging
